> 欢迎加入。读完本文（约 10 分钟）你会知道：这是什么项目、当前进展、你负责什么、从哪里开始、按什么规则协作。所有信息都在这个仓库里，本文是总入口。

## 1. 这是什么项目

把开源的 **Chatbox**（AI 聊天客户端，Electron + React + Capacitor）二次开发成一款**二次元向的 AI 伴侣安卓 App**：

- **多角色微信式**：App 像微信，每个 AI 角色像一个联系人，各自有人设、声音、表情包、记忆
- **核心体验**：双向语音消息（含音色克隆——用户给一段语音样本，AI 就能用那个声音说话）、双向表情包（AI 会挑表情发，也能看懂你发的）、长期记忆（自动记住关于你的事，有可管理的"记忆册"）
- **商业模式**：默认接自研后端（邮箱验证码登录 + 卡密充值 + 积分计费），同时保留用户自接任意 OpenAI 兼容中转站的能力
- **隐私立场**：零遥测（无任何统计/崩溃上报），对话与记忆本地优先——这是卖点

基线锁定上游 `chatbox v1.21.1`（tag `baseline-v1.21.1`），**不跟进上游更新**，需要官方修复时单独 cherry-pick。

## 6. 协作规则（5 条，都很轻但请遵守）

1. **不直推 main**：一律短分支 → PR → 对方 Approve → 合并。仓库是免费版私有仓，GitHub 不会强制拦你，全靠自觉
2. **每天一条同步消息**：昨天做了什么 / 今天做什么 / 卡在哪。不用开会
3. **决策进文档**：聊天里说定的任何事，改到 `docs/plans/` 对应文档里才算数（本项目所有决策都有编号，D1–D19，新决策续编号）
4. **计划可以质疑**：实施计划与实际代码冲突时，以代码现状为准，在 PR 里说明偏离原因；觉得方案不对，先提出讨论再动手，别默默改方向
5. **协议是合同**：`backend-api-spec.md` 定稿后，任何接口改动需要你和 Alex 双方确认并更新文档版本号

## 7. 常见问题

- **git 历史为什么这么短？** 基线是重建的无父根提交（上游是浅克隆推不上 GitHub），`baseline-v1.21.1` 这个 tag 就是 v1.21.1 快照，属正常
- **我会不会和别人改冲突？** 分工边界清晰：你主战场在 `src/renderer/`；`android/` 原生层归 Claude；后端在另一个仓库。重叠极少
- **代码库好大，从哪看起？** frontend-dev-guide §2 有导览图，只需要先懂：平台抽象（`src/renderer/platform/`）、消息 parts 模型（`src/shared/types/session.ts`）、路由（`src/renderer/routes/`）三件事
- **UI 库怎么混着 MUI、Mantine、Tailwind？** 历史原因，规则是"改哪个文件就跟哪个文件的风格"，不做统一重构
- **测试怎么跑？** `pnpm test`（vitest）；提交前必过 `pnpm run check`（tsc）+ `pnpm run lint`（biome）
- **遇到问题问谁？** 优先在 PR/issue 里留言（可追溯）；急事直接找 Alex；需要 AI 支援（查代码、解释架构、改计划）让 Alex 安排
