## 一、已确认决策

| # | 决策点 | 结论 |

| D1 | 总目标 | 跑通构建 → 品牌定制 → 自研后端为默认 → 伴侣功能增改，长期迭代 |
| D2 | 分发方式 | 第一阶段自分发 APK（不上商店），合规压力最低 |

| D8 | 产品形态 | 多角色微信式：会话列表=联系人列表，角色=Copilots 改造，每角色有人设/声音/表情包/记忆 |
| D9 | 记忆功能 | 基础=本地自动提取+记忆册（sqlite + LLM 事实提取，可查看/编辑/删除）；高级=云端 Honcho **托管版**（api.honcho.dev，经自研后端代理，opt-in） |
| D10 | 语音 | 微信式双向语音消息：用户录音→STT 转写气泡；AI 回复→TTS 语音气泡（每角色一个音色）；**支持音色克隆**：用户上传/播放语音样本→克隆出该音色→AI 用它语音沟通（fish-audio/MiniMax 类克隆 API；自测期客户端直连+自备 key，M3 起经自研后端代理）；STT 走后端 OpenAI 兼容 audio 接口 |
| D11 | 表情包 | 双向：用户导入建库 + 每角色专属表情包（用户帮角色配置，导入时视觉模型生成描述）；AI 经 `send_sticker` 工具发送；用户发的表情包走图片 part 由视觉模型理解；附少量无版权预置 |

## 二、代码库侦察结论（可复用 / 需新建）

**可复用：**
- 单 renderer 代码库，`CHATBOX_BUILD_TARGET=mobile_app` + `CHATBOX_BUILD_PLATFORM=android` 编译期切换；运行时 `src/renderer/platform/index.ts` 注入 `MobilePlatform`
- Copilots 角色系统（人设/头像/系统提示词/精选市场）→ 改造为"角色"
- parts 化消息模型 + 图片附件视觉管线（`message-parts/`）→ 用户发表情包直接复用
- blob 存储（`setBlob/getBlob`）→ 录音、表情包素材落地
- 工具调用框架（`packages/tools`）→ `send_sticker` 工具
- `@capacitor-community/sqlite`（现有会话存储在用）→ 本地记忆数据层
- 登录流代码（官方 ticket 登录）→ 改造为自研验证码登录
- `updateStore` / `mobile_logger` → 更新检查、本地日志排查

**需新建：**
- `capacitor.config.ts` + `android/` 原生工程（**官方未开源**，需 `npx cap add android` 自建并配置全部插件、图标、启动屏）
- `MemoryController` 接口 + 本地实现（sqlite+LLM 提取）+ Honcho 实现（后端代理）
- 语音消息 part 类型（消息模型/渲染/存储/上下文构建全链路，M2 最大风险项）
- 录音 UI + 音频播放气泡
- 表情包库管理（用户库/角色库、导入、描述生成）
- 自研默认 provider + 登录/卡密/积分余额 UI

**明确不依赖：** 移动端向量 RAG（知识库/附件 RAG 仅有 desktop-controller，移动端不存在；本地记忆不用向量，走结构化事实）

## 三、后端协议草案方向（M3 定稿）

- 推理面（OpenAI 兼容）：`/v1/chat/completions`（SSE 流式）、`/v1/audio/transcriptions`（STT）、`/v1/audio/speech`（TTS）、`/v1/embeddings`（预留）
- 音色面：`/voices` 管理（上传样本创建克隆音色 / 列表 / 删除），`/v1/audio/speech` 以 voice_id 引用克隆音色（后端对接 fish-audio/MiniMax 等克隆服务）
- 账号面：邮箱验证码（请求/校验）、卡密激活、积分余额查询、模型列表+单价下发、`/feedback` 意见反馈、`/app/version` 更新检查
- 高级记忆（M4）：Honcho 代理端点（客户端不持有 Honcho key；可作为付费能力）
- 预留：个人数据源（穿戴/健康）扩展位
- 设计约束：与 one-api/new-api 生态兼容，后端可自研也可拿开源网关改造

## 四、里程碑

| 阶段 | 内容 | 出口条件 |
|------|------|----------|
| M0 环境+基线 | JDK 21 + Android SDK + 模拟器（验 KVM）+ 真机调试；`pnpm install`；自建 capacitor.config + android/ 工程（占位包名）；接入全部 Capacitor 插件 | 从源码打出 APK，真机安装后现有功能冒烟通过 |
| M1 品牌+剥离 | 占位品牌名/图标/启动屏；剥离 Sentry/GA/Plausible/官方链接/评分弹窗；反馈入口占位 | APK 无任何官方上报流量与品牌痕迹 |
| M2 伴侣体验 | 微信式信息架构；角色扩展（声音/表情包库/记忆开关）；双向语音消息（预置音色起步，克隆音色直连 fish-audio 自测）；双向表情包；本地记忆+记忆册（自测用自接中转站） | 用中转站完整跑通"语音聊天（含克隆音色）+表情包+记忆"闭环 |
| M3 商业骨架 | 协议文档定稿；登录/卡密/积分 UI；自研默认 provider；语音克隆与 TTS 转经自研后端代理；应用内更新检查（后端可并行，客户端先 mock） | 后端就绪后即插即用；首个对外 APK（此前定死包名+生成并备份 keystore） |
| M4 高级记忆 | Honcho 经后端代理接入，opt-in + 隐私提示 | 本地/云端记忆可切换 |
| 后置 | 真远程推送（极光类）、悬浮球、穿戴/健康数据主动关怀、表情商店、RAG 记忆增强 | — |

## 五、待定项

- 正式品牌名 + 包名：二次元风格，用户尚未拍板，继续用占位包名开发（首个对外 APK 前定死）。已出候选：绊音 Kizune / 音灵 Otoria / 口袋纸片人 / 心匣 KokoBako（选定后需联网查重）
- 音色克隆供应商最终选型：倾向 fish-audio（克隆 API + 二次元社区音色市场，中文友好），备选 MiniMax
- 预置表情包素材来源（注意版权）

## 六、风险

1. **语音消息 = 新消息类型**，纵切消息模型/存储/渲染/上下文构建，是 M2 工作量与回归风险最大项。
2. **自建 android/ 工程与官方私有工程存在未知差异**（插件配置、混淆、权限），M0 冒烟要覆盖：流式请求（capacitor-stream-http）、sqlite、文件、键盘、安全区。
3. 模拟器依赖 KVM，不可用则降级为真机 only。
4. 零遥测意味着线上问题只能靠反馈+用户导出日志排查（已接受）。
5. **音色克隆的合规面**：上传声音样本需用户确认有权使用（App 内加授权声明与使用条款）；克隆知名声优/角色音色存在人格权与版权灰区，公开分发前需评估。
